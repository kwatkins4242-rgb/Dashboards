# ODIN Jarvis — Electron “Iron HUD” shell  11

One window tabs **HUD**, **browser**, **Swagger/API**, **Kali lab**, and an **embedded PTY** (same Chromium engine as Chrome, minus the bookmark bar circus).

Avoid opening `jarvis_mega.html` as `file://` if you rely on `:8000` APIs — **use this app** or `http://127.0.0.1:8000/`.

## Prerequisites

1. **Brain up**
   ```bash
   cd /home/kwatk42/jarvis
   .venv/bin/python main.py
   ```

2. **Fedora/Linux build toolchain** for `node-pty` (first time): `gcc`, `gcc-c++`, `make`, `python3-devel`, `npm`.

## Install & run

```bash
cd /home/kwatk42/jarvis/desktop/jarvis-odin-shell
npm install
npx electron-rebuild -f -w node-pty    # rerun after Electron majors upgrade
npm start
```

## Tabs & URL persistence

URLs are merged into Electron `userData/odin-hud-shell.json`:

| Tab | Stored key | Typical use |
|-----|-------------|-------------|
| **HUD** | `hudUrl` | `http://127.0.0.1:8000/` serves `jarvis_mega.html` via FastAPI |
| **BROWSER** | `browserUrl` | General browsing (fewer iframe limits than `<iframe>` HUD) |
| **SWAGGER / API** | `swaggerUrl` | OpenAPI / Swagger UI, extension docs — **whatever host:port serves your swagger** |
| **KALI LAB** | `labUrl` | e.g. **`ttyd` web shell**, Cockpit VM console, Jupyter on the VM |
| **TTY** | — | **`node-pty` + xterm.js** embedded local shell |

Use the **ACTIVE URL** bar plus **SAVE TO TAB** after you tune each tab once.

Bootstrap env overrides (applied before disk config merge):

```bash
export JARVIS_HUD_URL=http://127.0.0.1:8000/
export JARVIS_CRAWLER_HOME=https://duckduckgo.com/
export JARVIS_SWAGGER_URL=http://127.0.0.1:YOUR_PORT/swagger-ui/
export JARVIS_LAB_URL=http://127.0.0.1:7681/   # example ttyd
```

## Burp Suite

Burp Suite’s desktop UI **is not usually a localhost web page**. What you paste into **SWAGGER / API** is typically:

- a **Swagger / OpenAPI** server that documents an API behind Burp-testing, **or**
- any **Burp/extension HTTP panel** another tool exposes explicitly as a URL.

If you tell Cursor the actual URL your stack prints (often `localhost` + `/swagger*` or `/v3/api-docs`), mirror that string in **`JARVIS_SWAGGER_URL`** or the ACTIVE URL strip.

## Kali / tooling (no ISO download here)

Bundling multi‑GB ISOs inside this repo isn’t sane. Typical wiring:

| Pattern | HUD integration |
|---------|----------------|
| QEMU / virt-manager VM | SSH from **TTY**, or **`ttyd` → KALI LAB** tab |
| `docker run kalilinux/kali-rolling` | `docker exec -it … bash` in **TTY** |
| Dedicated laptop | **`labUrl`** pointed at **`ttyd`**, Cockpit, or Guacamole |

`/v1/mcp` plus **bridge :8099** stay the safer unattended agent surface versus “full root shell” autopilot.

## Menus still available

- **`JARVIS_TERMINAL_CMD`** — launches GNOME Terminal / Kitty *outside* the embedded PTY (`Panels` shortcut).

  CHANGES
  1. JARVIS_BRAIN_API_KEY (also BRAIN_API_KEY)
In .env:

Pick a long random secret (example):
openssl rand -hex 32
Set JARVIS_BRAIN_API_KEY=<that-string>.
Restart python main.py.
Behavior:

If the variable is empty → /v1/chat/completions is unchanged (no auth), good only for locked-down localhost.
If it is set → every POST /v1/chat/completions (and POST /v1/brain/simple) must include the secret in one of these ways:


curl -s -X POST 'http://127.0.0.1:8000/v1/chat/completions' \
  -H "Authorization: Bearer YOUR_SECRET_HERE" \
  -H "Content-Type: application/json" \
  -d '{"model":"4.7","messages":[{"role":"user","content":"Hi"}]}'
  
  
  curl -s -X POST 'http://127.0.0.1:8000/v1/chat/completions' \
  -H "X-JARVIS-BRAIN-KEY: YOUR_SECRET_HERE" \
  -H "Content-Type: application/json" \
  -d '{"model":"4.7","messages":[{"role":"user","content":"Hi"}]}'

Quick recap
Situation	What to do
Local only, nobody else on LAN
Leave JARVIS_BRAIN_API_KEY unset.
LAN / tunnel / VPS
Set JARVIS_BRAIN_API_KEY, always use Bearer or X-JARVIS-BRAIN-KEY.
Scripts / cron
Prefer POST /v1/brain/simple + Authorization: Bearer.
n8n via bridge
Set the key once in .env; bridge forwards it to the brain.
Charles’s rule applies: treat .env as secret; never commit JARVIS_BRAIN_API_KEY. If you want the same protections on /v1/memory or /screenshot next, say which routes and we can align them behind the same scheme.
